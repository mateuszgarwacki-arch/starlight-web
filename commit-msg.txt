S51: Suggested cut plan on traveller + extractor (2D/1D layout viz)

Visualises the bin-packing layouts already computed by the cutlist
extractor's guillotine and 1D-FFD algorithms — placements were
captured internally but discarded after sheet/length counting.

What changed:
- src/lib/cut-layout.ts (new) — shared types + pure calc functions
  (binPackLengths, guillotinePack, buildMaterialSummary, parseSheetSize,
  labelFor). Algorithms now return Placement[] and LengthBin[] capturing
  which piece goes where at what offset.
- src/components/cut-layout-renderer.tsx (new) — SVG renderers
  (SheetLayout, LengthBinRow, MaterialCutPlan, CutPlanSection). Compact
  mode: sheets 2-up at ~88mm, lengths 1-up at ~180mm. Dynamic font sizing
  per part. break-inside-avoid on each diagram + material block.
- src/app/api/extract-cutlist/route.ts — prompt updated to extract
  part_label from source (OpenCutList column, prefix, or null). No
  schema change; stored in existing extracted_data JSON.
- src/components/cutlist-extractor.tsx (rewritten) — calc moved to lib,
  layout toggle in both extracted+confirmed states, part_label column
  in parts reference table. Audit gap closed: tbl_wo_documents writes
  now go through auditedUpdate (3 sites: extract, confirm, re-add).
- src/app/traveller/page.tsx — full materials catalogue fetched once
  per traveller load, reused for both BOM enrichment and cut plan
  computation. WOData carries cutPlanSummaries: MaterialSummary[].
  TaskBrief renders <CutPlanSection> after the BOM table when
  summaries exist. Recompute, not persisted.
- docs/TRACKER.md — S51 entry + backlog item for remaining
  tbl_wo_bom raw insert gap.
