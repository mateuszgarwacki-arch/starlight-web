Tighten freelancer onboarding flow

Three issues stacked on top of each other:

1. Bug: PIN generator hardcoded to 4 digits.
   `Math.floor(1000 + Math.random() * 9000)` returned 1000-9999.
   Fixed to 6-digit (100000-999999).

2. Bug: Silent dead write on tbl_freelancers.pin column.
   The `pin` column doesn't exist on tbl_freelancers (verified via
   information_schema — 12 columns, none named `pin`). The savePin()
   handler wrote to it after the auth-sync API succeeded, but the
   error was never checked and the success toast already fired. So
   "PIN updated" appeared to work because Supabase Auth (the actual
   credential store) updated correctly. Removed the dead write and
   the matching phantom local-state spread.

3. Security: privilege-escalation foothold in freelancer-sync API.
   Role check fell back to user_metadata.role, which is user-editable
   per Supabase Auth model. SP-006 requires app_metadata only. Removed
   the fallback. No callable surface today exposed it, but defense in
   depth.

Plus consistency cleanup:
- Add Freelancer dialog: dropped the redundant PIN field. Onboarding
  is now one canonical path: Add → row appears → key icon →
  Mobile Access dialog → set PIN → WhatsApp invite. The Mobile
  Access dialog has the polish (Generate, eye toggle, validation,
  invitation message). Two competing PIN UIs collapsed to one.
- Mobile Access dialog: copy "PIN (4-6 digits)" → "PIN (6 digits)".
  Save button validation length >= 4 → length >= 6. Existing
  4-digit PINs continue to work (Supabase Auth has them hashed
  already); new and changed PINs must be 6 digits. Placeholder dot
  count updated.
- Mobile login: placeholder dot count updated to 6 dots.
- freelancer-sync API: accepts both `password` (new) and `pin`
  (deprecated, kept for back-compat) in body. crew page now sends
  `password`.

No schema changes. No RLS changes. Existing freelancer auth records
unchanged. Documentation gap (SP-002 in project knowledge claims
freelancer passwords are >= 8 chars; reality is 6-digit numeric PIN)
flagged separately for policy update.
