fix(jobs): back-to-job lands on Quote Lines, anchored to the line (S50e)

Mateusz: "i want to be taken to quote line tab - that's from where i operate."

Reframe of the original feature. S50a-d targeted "land on whatever tab
you came from" but the actual workflow is: PM lives on Quote Lines,
clicks into a WO or scope to inspect/edit, comes back. The anchor that
matters is the line, not the child element. A line can hold multiple
scopes and multiple WOs but they're all children of one line - landing
on the line gives the user immediate context for what they were just
looking at.

Changes:

1. Job page (renderLineRow) - quote line <tr> now carries
   id="line-{quote_line_id}" and scroll-mt-24 so it sits below the
   sticky topbar when scrolled into view.

2. Scope page (Back-to-Job link) - href is now always
   /jobs/X?tab=lines#line-{quote_line_id} (or just /jobs/X if the
   scope is orphaned from a line - the auto-overhead case). scroll={false}
   stays in place so Next.js doesn't fight the scrollIntoView.

3. Scroll-restore deps - lines.length added so the effect re-fires
   when the quote-lines data arrives.

Side effect: no more SSR tab-switching trap to worry about. The default
activeTab is "lines" and that matches both the server prerender and the
target tab. The useEffect that reads ?tab= still runs (harmless when t
is already "lines") but isn't the load-bearing path anymore.

The wo-{id} and scope-{id} anchors stay in place; they're no longer
reached by the back link but cost nothing and might be useful for
future deep-linking.
