fix: split material cost into Plan vs Actual everywhere

The Complete Job dialog showed Materials = £0 on a job with £305 of
allocated invoices because it relied on rpc_job_close_report's
material_cost_reconciled field, which sums BOM rows only — and that
job has no BOM. Same gap surfaced on the Job Close report.

Root cause: the system has two distinct concepts ("plan" via BOM,
"actual" via invoice allocations) but the close-report RPC was
collapsing them into a single misleadingly-named "reconciled" number.
"reconciled_line_cost" was structurally false anyway: it equals planned
in practice because nothing populates tbl_wo_bom.actual_unit_cost
automatically.

RPC changes (rpc_job_close_report):
- Replace material_cost_reconciled with material_cost_planned (BOM),
  material_cost_actual (invoice allocations), material_cost_committed
  (max of the two — drives margin per the system-wide rule).
- scope_cost: planned_material + actual_material per scope, allocations
  matched via scope_item_id OR via WO under the scope.
- material_by_category / material_by_supplier: BOM only (drop the bogus
  reconciled_cost). Actual breakdown by category/supplier deferred —
  needs invoice-line categorisation.

Dialog (job-complete-dialog.tsx):
- "Total Actual" → "Total Committed" (labour + max(plan,actual))
- Materials cell now shows "Plan £X / Actual £Y", red on overrun
- Unallocated-invoice warning text fixed: was "not yet allocated to BOM"
  (which the rollup never measured) — now correctly says "not allocated
  to a scope or work order"

Job Close report (/reports/job-close/[jobId]):
- Same Plan/Actual treatment in the commercial summary
- Material Cost by Scope: Plan / Actual / Variance columns with
  overrun highlighting
- Materials by Category / Spend by Supplier: labelled as "Planned",
  noted that actual-by-category requires invoice-line categorisation
- Same warning text fix
