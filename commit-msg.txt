fix(jobs): scroll-restore SSR hydration trap (S50d)

The real bug behind "url is correct but doesn't take me anywhere":

activeTab's useState lazy initialiser used `typeof window === "undefined"`
to read `?tab=` from the URL. This is a classic Next.js App Router
hydration trap:

- Server prerender: window undefined -> initialiser returns "lines"
- Server emits HTML with Quote Lines tab content rendered
- Client hydrates with the server's state value
- Even though the client-side initialiser would now return "wo", the
  hydrated state stays "lines"

Net effect: the URL has `?tab=wo#wo-XXX`, but the active tab is still
Quote Lines, so the WO rows are never rendered. document.querySelector
returns null, scrollIntoView never fires, page stays at top.

Fix: drop the window-check from the initialiser (now just "lines"),
and add a separate useEffect that reads `?tab=` after mount. This runs
purely client-side, post-hydration, and triggers a re-render to the
correct tab. The scroll-restore useEffect already has activeTab in its
deps, so it re-fires after the tab switch and finds the row.

Cost: a ~16ms flash of Quote Lines on initial load before the tab
switches. Acceptable; barely perceptible.

Same pattern (typeof window check in useState initialiser) exists for
activeFilter from sessionStorage. Leaving alone -- not the bug path and
the flash trade-off is uglier there.
