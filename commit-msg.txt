S37 BOM unit_cost invariant — fix triple-multiplier timber costing

Root cause: two coexisting conventions for unit_cost on Length rows
(per-metre vs per-length) with views, RPC, and frontend all applying
a standard_length/1000 multiplier on top. Timber rows displayed at
roughly 4.8x their correct total.

Database (migration s37_fix_bom_unit_cost_invariant):
- Normalised 6 legacy Convention A rows to Convention B
  (bom_ids 80, 88, 89, 90, 148, 182)
- qry_bom_enriched rebuilt: planned/reconciled line cost now simply
  qty * unit_cost. No multiplier.
- rpc_pm_job_overview rebuilt: same simplification in bom_raw CTE.
- Column comment on tbl_wo_bom.unit_cost and actual_unit_cost
  documenting the invariant.

Frontend (work-orders-panel.tsx):
- bomRowCost: dropped the standard_length multiplier.
- confirmScopeMaterial: completed the TODO metre->length conversion
  so new scope material rows write per-length cost when buyUnit=Length.
- Scope material preview: matching conversion so preview total agrees
  with the value that gets saved.
- Types: scopeMatSelected / scopeMatResults now carry standard_length.

Invariant (lock it in):
tbl_wo_bom.unit_cost is the price for ONE of tbl_wo_bom.unit. Line
total = quantity * unit_cost. No multipliers at read time. Any write
path that changes the stored unit must also convert the stored cost.

Verified on Grosvenor (job 8): bom_id 183 (3x1 PAR, 26m actual)
computes to GBP 31.68 (was GBP 152.06).
