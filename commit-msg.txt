Job Complete state — button, dialog, report, filters

Schema (already in DB from prior session):
- tbl_production_plan: completed_at, completed_by, close_note columns
- rpc_job_close_report(p_job_id INT) RETURNS json — full close summary
  with header, commercial, labour by freelancer + activity, materials by
  category + supplier, WO variance top movers, scope cost, learnings,
  post-complete edit count

This commit ships the UI:

Job page (src/app/(dashboard)/jobs/[id]/page.tsx):
- "Complete Job" button in toolbar (Active jobs)
- "Close Report" + "Reopen" buttons (Complete jobs)
- handleReopen handler: clears completed_at/by, keeps close_note as
  context, audit-logged
- JobCompleteDialog mounted alongside other dialogs

Job complete dialog (src/components/job-complete-dialog.tsx):
- Bug fixes from prior session: getAuditContext takes supabase, and
  completed_by resolved from JWT user_metadata (NULL for admin without
  a freelancer row)
- Loads rpc_job_close_report on open for live commercial summary
- Soft warning if any non-overhead WOs still active (does not block —
  soft signals principle)
- Optional close_note textarea, "Skip & Complete" button when empty
- Audit-logged update via auditedUpdate, conflict-aware
- On success: navigates to /reports/job-close/[jobId]

Close report (src/app/(dashboard)/reports/job-close/[jobId]/page.tsx):
- Live data — never a snapshot, always current
- Renders for any job; "Interim" badge if not yet Complete
- Post-complete edits warning when audit shows mutations after close
- Margin colour-coded (green >= 25%, amber >= 10%, red below)
- Print-friendly via existing report patterns

Filter Complete jobs from active surfaces:
- /jobs default list hides Complete; toggle "Show N complete" appears
  when any exist; pill colour distinguishes Complete (muted) from
  Active (green)
- /m mobile task list: fixed legacy filter that referenced "Closed"
  status (which never existed in this schema, so was a no-op) — now
  correctly filters Complete; both active and complete WO queries
  scoped to non-Complete jobs
- /capacity/add-booking: job picker excludes Complete (you don't book
  fresh crew onto finished jobs)
- /workshop and /capacity main pages: already filtered via
  rpc_workshop_data and rpc_capacity_data (prior session)
- Dashboard upcoming jobs: already filtered via rpc_dashboard_data
  (prior session)
- /m/task and /m/request: already filter to Active-only (existing)

DB views patched to ignore Complete jobs:
- qry_procurement_needed: had no job-status filter; was surfacing
  unordered BOM rows from finished jobs
- qry_stale_travellers: filtered Complete WOs but not Complete jobs;
  edge case where PM marks job Complete with WOs still Ready

No RLS changes. No tbl_audit_log changes (registry already had
tbl_production_plan from prior sessions).
